# ADR 014 — Installation ID

**Status:** Accepted  
**Date:** 2026-04-16

---

## Context

Several backend features require a stable, device-scoped identifier that
persists for the lifetime of an app install and is independent of:

- The user's identity (`userId` / `anonymousId`) — which changes on logout or
  profile switch
- The push token — which can rotate without any user action and may differ
  between push infrastructures (APNs vs FCM)

The immediate drivers are:

1. **Live Activity server integration** — The partner-facing Live Activity
   server needs to correlate regular push token records (held by the main push
   backend) with Live Activity push token records (registered by ActivityKit).
   Without a shared stable key, the server cannot perform the join needed to
   trigger the silent push start path.

2. **Push-to-start device targeting** — A partner sending a push-to-start
   request for a specific device (rather than all devices for a user) needs a
   device-scoped identifier to express that intent. The push token cannot serve
   this role because it rotates, and `userId` is too coarse (a user may have
   multiple devices).

3. **Cross-module device correlation** — As more modules (push, live
   activities, geofencing, location) register device-level data with the
   backend, a shared stable key makes backend-side joins reliable.

The `anonymousId` was considered as a proxy for this role — it is stable,
persisted, and already included on all events. It was rejected because its
semantic purpose is session/profile identity (anonymous user tracking), not
device identity. Reusing it would conflate two distinct concepts, make future
renaming or behavioural changes risky, and expose a confusing API surface to
partners who see `installationId` in Live Activity payloads and expect it to
mean something device-specific.

---

## Decision

The SDK generates a dedicated `installationId` UUID on first launch and
persists it for the lifetime of the app install.

### Generation and storage

- Generated as a `UUID` on first access if no value is stored.
- Persisted in `StorageManager` (SqlCipher-encrypted on-device database).
- Cleared and regenerated by `CustomerIO.reset()` alongside `anonymousId`.
- Survives app updates, SDK updates, and user identity changes that do not call `reset()`.

### Exposure

- Exposed as a `nonisolated` read-only property on the root `CustomerIO`
  actor so any module can read it without an `await`.

```swift
extension CustomerIO {
    public nonisolated var installationId: String { … }
}
```

### Inclusion in push registration

`MessagingPushModule.emitDeviceCreatedOrUpdated()` includes `installationId`
in the `"Device Created or Updated"` event payload. The backend stores it
alongside the push token so that the Live Activity server (once integrated)
can join on this value.

### Inclusion in Live Activity token registration

All SDK-surface Live Activity token registration calls include `installationId`
in the request body:

- `PUT /v1/live_activity_push_to_start/{activity_type}`
- `PUT /v1/live_activities/{activity_id}/push_token`

This allows the Live Activity server to resolve:

```
(userId, installationId, activityType) → push-to-start token
(userId, installationId, activityType) → instance push token
```

and enables partners to target push-to-start events at a specific installation
rather than fanning out to all registered devices for a user.

### Relationship to `anonymousId`

`installationId` and `anonymousId` are independent values that may coincide
in value on a fresh install (both are newly generated UUIDs) but are stored,
managed, and semantically distinct:

| | `anonymousId` | `installationId` |
|---|---|---|
| **Purpose** | Anonymous user / session identity | Device / install identity |
| **Reset on** | `CustomerIO.reset()` | `CustomerIO.reset()` or app deletion |
| **Included on** | All events (via `EventEnricher`) | Device registration + Live Activity registration |
| **Exposed as** | Internal SDK concept | Public read-only property on `cio` |

## Consequences

### Enables

- Silent push start path for Live Activities: the integrated push backend can
  look up a device's regular push token by `(userId, installationId)` and send
  a background push to trigger in-app Live Activity creation.
- Single-device push-to-start targeting: partners supply `installationId` on
  the start endpoint to target one specific installation.
- Reliable cross-module device correlation on the backend without depending on
  push token stability.

### Constrains

- `installationId` is only reliably readable after `configure()` completes
  (database access required). All current code paths that need it — push token
  registration events and Live Activity token registration calls — run after
  configure, so this is not a practical constraint today. See OPEN_QUESTIONS.md
  §6 for the pre-existing APNs token delivery race that is the closest related
  concern.
- Whether `installationId` should be stamped on all general tracking events
  (as `anonymousId` is) is deferred — see OPEN_QUESTIONS.md §5.
- Whether to make push transport (`apns` vs `fcm`) explicit in the device
  registration payload is also deferred — the backend currently infers this
  from token format, which is sufficient for now.

Nothing. This introduces a new concept; it does not replace an existing ADR.
