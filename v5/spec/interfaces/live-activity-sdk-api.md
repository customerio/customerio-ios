# Live Activity ‚Äî SDK API

> **Status:** Draft ‚Äî endpoint paths provisional pending backend confirmation.
>
> This document describes the HTTP surface the Customer.io iOS SDK calls on
> behalf of the host app during Live Activity and Live Update lifecycle
> management. Host apps never call these endpoints directly.
>
> For the partner-facing surface (start / update / end campaigns, backend data
> model, push payload structures), see
> [`live-activity-partner-api.md`](live-activity-partner-api.md).

---

## Platform Overview

### iOS ‚Äî Live Activities (iOS 17.2+)

ActivityKit manages live activities as first-class OS objects. APNs natively
understands the Live Activity push type and can deliver updates directly to a
running activity even when the host app is suspended or killed. A separate
push-to-start token (distinct from the regular device push token) allows the
server to remotely launch a new activity on a locked device.

### Android ‚Äî Live Updates (Android 16 / API 36+)

Live Updates are built on top of the standard ongoing notification system.
Android does **not** issue a separate token for live updates ‚Äî the same FCM
registration token used for all push notifications is used here. The SDK must
be alive to receive the FCM data message, decode the content state, and call
`NotificationManager.notify()` with the same notification ID to update the
existing live update in place. There is no push-to-start equivalent on Android.

## Common Conventions

| Convention | Detail |
|---|---|
| Base URL | Configured at SDK init via `LiveActivityConfigBuilder(baseURL:)`. Will be derived from `SdkConfig.region` once backend paths are finalised. |
| Request body | `application/json` |
| Response body | `application/json` |
| Auth | HTTP Basic ‚Äî `cdpApiKey` as username, empty password: `Authorization: Basic <base64("cdpApiKey:")>` |
| `activityInstanceId` | The partner-supplied identifier for a running live activity instance, taken from `CIOActivityAttribute.activityInstanceId`. Used in all instance-level routes. The partner sets this to a value meaningful in their own domain (e.g. an order ID). The SDK percent-encodes it when embedding it in URL paths. Must be stable for the activity's lifetime and globally unique within the Customer.io site. |
| `activityId` | The opaque ID assigned by ActivityKit (`Activity<T>.id`). Exposed in `LiveActivityInfo.activityId` for local ActivityKit correlation. **Not used in server routes** ‚Äî all instance-level routes use `activityInstanceId`. |
| `activityType` | A stable reverse-DNS string identifier declared in `register(_:identifier:)` (e.g. `io.yourapp.liveactivities.order`). Used in push-to-start registration routes and matched server-side to route start requests. Must not change after shipping. |
| `userId` | The identifier supplied to `CustomerIO.identify()`, or the anonymous ID if identify has not been called. |
| `installationId` | A stable UUID generated by the SDK on first launch and persisted for the lifetime of the app install. Uniquely identifies a device independently of user identity or push token. |
| `os` | Device operating system. `"ios"` or `"android"`. Tells the server whether a push token is expected and which delivery infrastructure to use. |
| `transport` | Push delivery infrastructure. `"apns"` for direct APNs delivery; `"fcm"` for Firebase Cloud Messaging relay. Determines which credentials the server uses. |
| Token format | Hex-encoded lowercase string for APNs tokens; URL-safe base64 string for FCM registration tokens. |

**Valid `os` / `transport` combinations**

| `os` | `transport` | When to use |
|---|---|---|
| `"ios"` | `"apns"` | Partner's APNs credentials are configured directly with Customer.io. |
| `"ios"` | `"fcm"` | Partner uses Firebase for iOS push; APNs credentials live with Firebase. ActivityKit-generated Live Activity tokens are still raw APNs tokens but can be relayed through FCM using Firebase's Live Activity API. |
| `"android"` | `"fcm"` | All Android devices. FCM is the only supported transport on Android. |
| `"android"` | `"apns"` | Invalid. |

## Retry Behaviour

| Response class | Action |
|---|---|
| `2xx` | Success ‚Äî no retry. |
| `4xx` | Log error ‚Äî no retry. |
| `5xx` or network error | Retry with exponential backoff, up to 3 attempts. If the activity ends before a retry succeeds, the pending retry is cancelled. |

## Endpoints

### `GET /health`

Liveness check. The SDK may call this during configuration to verify
connectivity to the Live Activity backend.

**Response ‚Äî 200 OK**

```json
{ "status": "ok" }
```

---

### `PUT /v1/live_activities/registration/{activityType}`

Register or rotate the **push-to-start token** for an activity type.

**iOS:** Called whenever ActivityKit vends a new push-to-start token for a
registered type. The token is required. The server stores the most recent token
per `(userId, installationId, activityType)` triple; older tokens are silently
replaced.

**Android:** Called at configure time for each activity type registered in the
SDK configuration. No token is included ‚Äî the FCM registration token is already
on file from push registration. The call records `(userId, installationId, activityType)`
so the server can route partner-initiated start requests to the correct installations.

**Path parameters**

| Parameter | Type | Description |
|---|---|---|
| `activityType` | string | Stable reverse-DNS identifier declared in `register(_:identifier:)`. URL-encoded. |

**Request body (iOS)**

```json
{
  "token": "lowercase_hex_push_to_start_token",
  "os": "ios",
  "transport": "apns",
  "userId": "user-id-or-anonymous-id",
  "installationId": "stable-install-uuid"
}
```

**Request body (Android)**

```json
{
  "os": "android",
  "transport": "fcm",
  "userId": "user-id-or-anonymous-id",
  "installationId": "stable-install-uuid"
}
```

| Field | Required | Description |
|---|---|---|
| `token` | iOS only | Hex-encoded APNs push-to-start token. ActivityKit generates this directly regardless of the partner's push configuration. Omitted for Android. |
| `os` | Yes | `"ios"` or `"android"`. |
| `transport` | Yes | `"apns"` or `"fcm"`. For Android, always `"fcm"`. |
| `userId` | Yes | Identified user ID, or anonymous ID. |
| `installationId` | Yes | Stable per-install UUID. |

> **Server note ‚Äî FCM transport validation:** When `transport` is `"fcm"`, the
> server must verify that a valid FCM registration token exists for the device
> identified by `installationId`. If no FCM token is on file, respond with 4xx.
> The SDK will not retry 4xx responses.

**Response ‚Äî 200 OK**

```json
{ "registered": true }
```

**Response ‚Äî 400 Bad Request**

```json
{ "error": "<missing field name> is required" }
```

### `PUT /v1/live_activities/{activityInstanceId}/push_token`

Register or rotate the **instance push token** for a specific live activity.
This is the token the server uses to deliver content-state updates and end
events to a single running activity instance.

On iOS this is the ActivityKit instance token, distinct from both the device's
regular push token and the push-to-start token. On Android this is the standard
FCM registration token.

The SDK calls this when an activity starts (initial token) and whenever the
token rotates for a running activity.

**Path parameters**

| Parameter | Type | Description |
|---|---|---|
| `activityInstanceId` | string | Partner-supplied activity identifier (`CIOActivityAttribute.activityInstanceId`), percent-encoded. |

**Request body (iOS)**

```json
{
  "token": "lowercase_hex_instance_token",
  "activity_type": "io.yourapp.liveactivities.order",
  "os": "ios",
  "transport": "apns",
  "userId": "user-id-or-anonymous-id",
  "installationId": "stable-install-uuid"
}
```

**Request body (Android)**

```json
{
  "activity_type": "io.yourapp.liveactivities.order",
  "os": "android",
  "transport": "fcm",
  "userId": "user-id-or-anonymous-id",
  "installationId": "stable-install-uuid"
}
```

| Field | Required | Description |
|---|---|---|
| `token` | iOS only | Hex-encoded APNs instance push token. Omitted for Android (FCM token already on file). |
| `activity_type` | Yes | Stable reverse-DNS identifier for this template. |
| `os` | Yes | `"ios"` or `"android"`. |
| `transport` | Yes | `"apns"` or `"fcm"`. See valid combinations above. |
| `userId` | Yes | Identified user ID, or anonymous ID. |
| `installationId` | Yes | Stable per-install UUID. |

> **Server note ‚Äî FCM transport validation:** Same as push-to-start ‚Äî verify
> an FCM token exists for `installationId` when `transport` is `"fcm"`.

**Response ‚Äî 200 OK**

```json
{ "registered": true }
```

**Response ‚Äî 400 Bad Request**

```json
{ "error": "<missing field name> is required" }
```

### `PUT /v1/live_activities/{activityInstanceId}`

Content-state update acknowledgement. The SDK calls this when the host app
updates a live activity's content state locally (i.e. from within the app,
not via push). The server records the new state for auditing but does not
act on it ‚Äî the SDK has already applied the change via the platform's native API.

**Path parameters**

| Parameter | Type | Description |
|---|---|---|
| `activityInstanceId` | string | Partner-supplied activity identifier, percent-encoded. |

**Request body**

```json
{
  "contentState": { "<template-specific content-state fields>" }
}
```

| Field | Required | Description |
|---|---|---|
| `contentState` | Yes | Updated content-state object. Schema is template-specific ‚Äî see [Content State Schemas](live-activity-partner-api.md#content-state-schemas) in the Partner API doc. Note the camelCase field name. |

The user association is established at instance push token registration and is
not repeated here.

**Response ‚Äî 200 OK**

```json
{ "ok": true }
```

---

### `DELETE /v1/live_activities/{activityInstanceId}`

Activity ended notification. The SDK calls this when a live activity reaches
its terminal state (ended by the host app, ended by the server, or dismissed
by the system). The server removes the activity's push token from its registry.

**Path parameters**

| Parameter | Type | Description |
|---|---|---|
| `activityInstanceId` | string | Partner-supplied activity identifier, percent-encoded. |

**Request body**

```json
{}
```

> **Pre-release:** A `reason` field indicating the terminal state
> (`"ended"`, `"dismissed"`, or `"stale"`) is a strong candidate to add before
> shipping ‚Äî it could drive server-side analytics or suppression logic. Needs
> backend agreement before finalising.

**Response ‚Äî 200 OK**

```json
{ "ok": true }
```

The server must return `200` even if the activity was already ended (idempotent).
