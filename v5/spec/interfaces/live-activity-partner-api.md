# Live Activity — Partner API

> **Status:** Draft — Partner surface is in active design.
>
> This document describes the HTTP surface a partner's backend calls to start,
> update, and end live activities on behalf of their users. It also covers the
> push payload structures the CIO server constructs internally when dispatching
> to APNs or FCM, the backend data model, and backend operational guidance.
>
> For the SDK-facing surface (token registration, content-state acknowledgement,
> endpoint conventions), see
> [`live-activity-sdk-api.md`](live-activity-sdk-api.md).

---

## Platform Overview

### iOS — Live Activities (iOS 17.2+)

ActivityKit manages live activities as first-class OS objects. APNs natively
understands the Live Activity push type and can deliver updates directly to a
running activity even when the host app is suspended or killed. A separate
push-to-start token (per activity type, not per instance) allows the server to
remotely launch a new activity on a locked device.

### Android — Live Updates (Android 16 / API 36+)

Live Updates are built on top of the standard ongoing notification system.
Android does **not** issue a separate token — the same FCM registration token
used for all push notifications is used here. Updates are delivered as FCM data
messages; the SDK wakes, decodes the content state, and calls
`NotificationManager.notify()` with the same notification ID to update the
existing live update in place. There is no push-to-start equivalent.

## Common Conventions

| Convention | Detail |
|---|---|
| Base URL | Configured per deployment. |
| Request body | `application/json` |
| Response body | `application/json` |
| Auth | Bearer token auth expected in production. POC server has no auth on the partner surface. |
| `userId` | The identifier the host app supplied to `CustomerIO.identify()`. Partners target users by this ID. |
| `activityType` | Stable reverse-DNS string (e.g. `io.yourapp.liveactivities.order`). Registered by the app at SDK config time and stored server-side when the first push-to-start token is received. Must not change after shipping — campaigns reference this string. |
| `activityInstanceId` | Partner-supplied stable identifier for a specific running activity instance. Set by the app on the `CIOActivityAttribute.activityInstanceId` property before starting the activity. Partners use this to narrow update / end requests to a specific instance. |
| `installationId` | Stable per-install UUID generated by the SDK. Partners may use this to narrow requests to a specific device. |

## Partner Surface

Partners interact with live activities through user identifiers. The server
maintains the mapping from user identifiers to platform push tokens internally;
partners never handle tokens directly, and platform differences (APNs vs FCM)
are transparent to the partner.

All partner endpoints require `userId` and `activityType`. An optional
`activityInstanceId` can be supplied to narrow the operation to a single
running activity. When omitted, the operation applies to all matching activities
for that user and type across all platforms.

---

### `POST /v1/users/{userId}/live_activities/{activityType}/update`

Sends a content-state **update** to one or all running activities of the given
type for this user. The server dispatches via APNs for iOS and via FCM for
Android; the partner payload is identical in both cases.

**Path parameters**

| Parameter | Type | Description |
|---|---|---|
| `userId` | string | The user identifier supplied to `CustomerIO.identify()`. |
| `activityType` | string | Stable reverse-DNS activity type identifier. |

**Request body**

```json
{
  "activityInstanceId": "optional-activity-instance-id",
  "installationId": "optional-installation-id",
  "contentState": {
    "<template-specific content-state fields>"
  }
}
```

| Field | Required | Description |
|---|---|---|
| `activityInstanceId` | No | Narrows the update to the activity instance with this ID. |
| `installationId` | No | Narrows the update to a specific installation. |
| `contentState` | Yes | The new content state. Schema is template-specific — see [Content State Schemas](#content-state-schemas). |

When neither `activityInstanceId` nor `installationId` is provided, all
matching activities for this user and type receive the update. When both are
provided, the target must match both.

**Response — 200 OK**

```json
{
  "sent": 2,
  "failed": 0,
  "errors": []
}
```

| Field | Type | Description |
|---|---|---|
| `sent` | int | Number of activity instances the update was delivered to (APNs or FCM returned success). |
| `failed` | int | Number of delivery failures. |
| `errors` | array | Detail for each failure. |

### `POST /v1/users/{userId}/live_activities/{activityType}/end`

Sends an **end** event to one or all running activities of the given type for
this user, then removes their tokens from the registry.

On iOS the APNs end event causes the system to dismiss the activity from the
Lock Screen after `dismissalDate`. On Android the SDK processes the FCM data
message and cancels the ongoing notification.

**Path parameters**

| Parameter | Type | Description |
|---|---|---|
| `userId` | string | The user identifier. |
| `activityType` | string | Stable reverse-DNS activity type identifier. |

**Request body**

```json
{
  "activityInstanceId": "optional-activity-instance-id",
  "installationId": "optional-installation-id",
  "contentState": {
    "<final content-state fields>"
  },
  "dismissalDate": 1713100030
}
```

| Field | Required | Description |
|---|---|---|
| `activityInstanceId` | No | Narrows the end request to the matching activity instance. |
| `installationId` | No | Narrows the end request to a specific installation. |
| `contentState` | No | Final content state shown before the activity is dismissed. Defaults to the last-known state. |
| `dismissalDate` | No | **iOS only.** Unix timestamp after which the system removes the activity from the Lock Screen. Defaults to `now + 30 s`. |

**Response — 200 OK**

Same shape as the update response, with an additional `"purged"` count
reflecting how many tokens were removed from the registry.

### `POST /v1/users/{userId}/live_activities/{activityType}/start`

Requests that a new live activity be started for this user.

#### iOS — two start paths

**Preferred path — silent push.** The server sends a silent (background) push
to the user's registered device token via the device's configured transport
(`apns` or `fcm`). The app wakes in the background, creates the activity
locally via ActivityKit, and the resulting instance push token flows back to
the server via the SDK surface.

The silent push path requires the integrated push backend to have the device's
regular push token on record alongside `installationId`, so the server can
resolve `userId + installationId → (token, transport)` and dispatch correctly.
This join is not available in the standalone POC server.

**Fallback path — push-to-start.** When `installationId` is provided, the
server uses the APNs push-to-start token registered for that
`(userId, installationId, activityType)` triple to remotely launch the activity
without the app needing to be in the foreground. If `installationId` is
omitted, the server fans out to all push-to-start tokens registered for this
user and type. Push-to-start is best-effort — iOS may not grant background
execution time, and the token may not be registered if the user has not opened
the app recently.

**POC limitation:** The standalone POC server does not have access to the
regular APNs or FCM device tokens used for the silent push start path. The POC
exercises the **push-to-start path only** for iOS start requests and cannot
exercise the Android start path at all. The production implementation will use
the silent push path as primary once the live activity server is integrated into
the push backend.

#### Android — in-app start only

Android has no push-to-start equivalent. The server sends a high-priority FCM
data message signalling the SDK to create a live update. A force-stopped app
will not be woken. The silent push path is the only supported start mechanism
on Android.

**Path parameters**

| Parameter | Type | Description |
|---|---|---|
| `userId` | string | The user identifier. |
| `activityType` | string | Stable reverse-DNS activity type identifier. |

**Request body**

```json
{
  "installationId": "optional-installation-id",
  "attributes": {
    "<template-specific static attribute fields>"
  },
  "contentState": {
    "<template-specific initial content-state fields>"
  }
}
```

| Field | Required | Description |
|---|---|---|
| `installationId` | No | When present, targets the start request to a specific installation. When absent, all registered installations for this user and activity type are targeted. |
| `attributes` | Yes | Static attributes set at activity creation. Cannot be changed after the activity starts. Schema is template-specific — see [Attribute Schemas](#attribute-schemas). |
| `contentState` | Yes | Initial content state. Schema is template-specific — see [Content State Schemas](#content-state-schemas). |

**Response — 200 OK**

```json
{
  "sent": 1,
  "failed": 0,
  "errors": []
}
```

## Push Payload Structures

The server constructs platform-specific push payloads internally. This section
documents what the server sends to APNs or FCM. Partners do not send these
payloads directly.

### APNs — update / end (iOS)

```json
{
  "aps": {
    "timestamp": 1713100000,
    "event": "update",
    "content-state": {
      "<template-specific content-state fields>"
    },
    "alert": {
      "title": "Optional alert title",
      "body": "Optional alert body"
    },
    "dismissal-date": 1713100030
  }
}
```

`event` is `"update"` or `"end"`. `alert` and `dismissal-date` are only
included when explicitly supplied by the partner. Including `alert` in Live
Activity pushes should be avoided unless there is a specific UX reason to show
a banner — see [Backend Behavior Suggestions](#backend-behavior-suggestions).

APNs headers required:
```
apns-push-type: liveactivity
apns-topic:     {bundle-id}.push-type.liveactivity
apns-priority:  5   (silent, preferred)
                10  (alert — use sparingly)
```

### APNs — push-to-start (iOS 17.2+)

```json
{
  "aps": {
    "timestamp": 1713100000,
    "event": "start",
    "attributes-type": "OrderActivityAttributes",
    "attributes": {
      "<template-specific static attribute fields>"
    },
    "content-state": {
      "<template-specific initial content-state fields>"
    }
  }
}
```

`attributes-type` must exactly match the Swift type name of the app's
`ActivityAttributes` conformance (case-sensitive). See
[Backend Behavior Suggestions](#backend-behavior-suggestions) for the
fragility implications.

### FCM data message — update / end / start (Android)

```json
{
  "message": {
    "token": "<fcm registration token>",
    "data": {
      "event": "update",
      "activity_instance_id": "<activity-instance-id>",
      "activity_type": "io.yourapp.liveactivities.order",
      "content_state": "<JSON-encoded content-state object>"
    }
  }
}
```

`event` is `"start"`, `"update"`, or `"end"`. For `"start"`, `data` also
includes `"attributes"` (JSON-encoded static attribute object). The SDK decodes
`content_state` (and `attributes` on start), applies the update to the ongoing
notification via `NotificationManager.notify()`, and dismisses on `"end"`.

## Backend Data Model

```sql
-- Per-activity instance push tokens (iOS)
CREATE TABLE live_activity_tokens (
    workspace_id     TEXT        NOT NULL,
    device_id        TEXT        NOT NULL,
    profile_id       TEXT,                    -- NULL if anonymous at time of registration
    activity_id      TEXT        NOT NULL,    -- activityInstanceId from CIOActivityAttribute
    activity_type    TEXT        NOT NULL,
    token            TEXT        NOT NULL,
    attributes_type  TEXT,                    -- Swift type name; required for push-to-start reply
    attributes       JSONB,
    status           TEXT        NOT NULL DEFAULT 'active',   -- 'active' | 'ended'
    end_reason       TEXT,                    -- populated when status = 'ended'
    registered_at    TIMESTAMPTZ NOT NULL,
    token_updated_at TIMESTAMPTZ NOT NULL,
    ended_at         TIMESTAMPTZ,
    PRIMARY KEY (workspace_id, activity_id)
);

-- Push-to-start tokens (iOS 17.2+)
-- One row per (workspace, device, activity_type). Upserted on each token update.
CREATE TABLE live_activity_push_to_start_tokens (
    workspace_id     TEXT        NOT NULL,
    device_id        TEXT        NOT NULL,
    profile_id       TEXT,
    activity_type    TEXT        NOT NULL,
    token            TEXT        NOT NULL,
    registered_at    TIMESTAMPTZ NOT NULL,
    token_updated_at TIMESTAMPTZ NOT NULL,
    PRIMARY KEY (workspace_id, device_id, activity_type)
);
```

## Campaign Endpoint

Called by the CIO campaign engine when a marketer or automation triggers a Live
Activity action. This is an internal endpoint, not exposed to partners directly.

**`POST /api/campaigns/v1/live_activity/send`**

```json
{
  "profile_id":      "prof_abc123",
  "activity_id":     "act_xyz",          // required for "update" and "end" actions
  "activity_type":   "io.yourapp.liveactivities.order",  // required for "start" action
  "action":          "update",           // "start" | "update" | "end"

  "content_state":   { },                // for "update" and "start"; opaque pass-through
  "alert": {                             // optional; alert-priority updates only
    "title": "Order updated",
    "body":  "Your order is almost ready"
  },
  "apns_priority":   5,                  // 5 (silent) or 10 (alert); default 5
  "stale_date":      1711753600,         // optional Unix timestamp

  "dismissal_date":  1711750300,         // "end" only; optional Unix timestamp

  "attributes":      { },                // "start" only; initial ActivityAttributes values
  "attributes_type": "OrderActivityAttributes"  // "start" only; Swift type name
}
```

The backend resolves the target token from `live_activity_tokens` (update/end)
or `live_activity_push_to_start_tokens` (start) using `profile_id + activity_id`
/ `activity_type`. If no token exists or `status = 'ended'`, the send is a
no-op and the campaign records a "not deliverable" outcome.

For Android, the backend determines the platform from the device record and
uses the FCM device token with the Live Update payload format.

## Backend Behavior Suggestions

These items are handled by the backend and do not require SDK changes, but are
included here to document expected server behavior that affects overall system
correctness.

**APNs key management.** The `apns-push-type: liveactivity` push type uses the
same `.p8` key as standard `alert` pushes. The `apns-topic` suffix differs
(`{bundle-id}.push-type.liveactivity` vs `{bundle-id}`). No separate key is
required, but the backend must use the correct topic for each push type.

**Stale token cleanup.** Tokens for ended activities (`status = 'ended'`) must
not be pushed to. When APNs returns `410 Gone`, mark the record `status = 'ended'`.
Cleanup of rows older than a configurable retention window (suggested: 30 days
post-end) is a backend hygiene task.

**Token deduplication.** A PUT with the same token value as the current record
is a no-op — `token_updated_at` must not be updated. Only update the timestamp
when the token value changes.

**Profile token ownership transfer.** Live activity tokens registered while a
device is anonymous (`profile_id = NULL`) must be reassigned to the identified
profile when the SDK calls `identify()`. This mirrors existing behaviour for
device push tokens and ensures activities started before login can be targeted
by profile-based campaigns.

**`attributes-type` string management (push-to-start).** The `attributes-type`
field in APNs push-to-start payloads must exactly match the Swift type name of
the app's `ActivityAttributes` conformance. This string is stored when the SDK
registers a push-to-start token. If the app renames its `ActivityAttributes`
type, the stored string becomes stale and push-to-start will silently fail.

The stable `activityType` string (the reverse-DNS identifier declared in
`register(_:identifier:)`) is the reliable identifier for backend targeting and
campaign authoring. `attributes-type` (the Swift type name) is only needed for
the APNs push-to-start payload and should be sourced from the stored
push-to-start token record, not entered manually in campaign configuration.

**Alert banners alongside Live Activity updates.** If an `alert` key is
included in a Live Activity push payload and the user taps the resulting banner,
the OS delivers the tap through `userNotificationCenter(_:didReceive:)` — the
same delegate method `MessagingPushModule` monitors for standard push taps.
`MessagingPushModule` guards against this by checking for the `aps.event` field
(present in all Live Activity payloads, absent from standard CIO push payloads)
and exiting early. Avoid including `alert` in Live Activity pushes unless there
is a specific UX reason — keeping the payload types separate eliminates this
edge case entirely.
