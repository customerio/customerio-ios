# Live Activity Server API

> **Status:** Draft — Partner surface is in active design.
>
> This document describes the two HTTP surfaces exposed by the Live Activity
> server:
>
> 1. **SDK Surface** — endpoints called by the Customer.io iOS or Android SDK
>    on behalf of the host app. The SDK handles all calls on this surface
>    automatically; integrators do not call these endpoints directly.
>
> 2. **Partner Surface** — endpoints called by a partner's backend to start,
>    update, and end live activities on behalf of their users.

---

## Platform Overview

Live activity features are available on two platforms with meaningfully
different architectures. The server must account for both.

### iOS — Live Activities (iOS 17.2+)

ActivityKit manages live activities as first-class OS objects. APNs natively
understands the Live Activity push type and can deliver updates directly to a
running activity even when the host app is suspended or killed. A separate
push-to-start token (distinct from the regular device push token) allows the
server to remotely launch a new activity on a locked device.

### Android — Live Updates (Android 16 / API 36+)

Live Updates are built on top of the standard ongoing notification system.
Android does **not** issue a separate token for live updates — the same FCM
registration token used for all push notifications is used here. Updates are
delivered as standard FCM data messages; the SDK must be alive to receive the
message, decode the content state, and call `NotificationManager.notify()` with
the same notification ID to update the existing live update in place. There is
no push-to-start equivalent on Android — the app must already be running to
start a live update.

## Common Conventions

| Convention | Detail |
|---|---|
| Base URL | Configured at SDK init. |
| Request body | `application/json` |
| Response body | `application/json` |
| Auth | None in the POC server. Production deployments are expected to add bearer token auth on the partner surface. |
| Activity Instance ID | The partner-supplied identifier for a running live activity instance, taken from `CIOActivityAttribute.activityInstanceId`. Used in all SDK surface routes that target a specific activity instance (`/push_token`, content-state update, end). The partner sets this to a value meaningful in their own domain (e.g. an order ID) so that Customer.io webhook callbacks map directly to their entities. The SDK percent-encodes it when placing it in URL paths. Must be stable for the activity's lifetime and globally unique within the Customer.io site (identified by `cdpApiKey`). |
| Activity ID | The opaque ID assigned by ActivityKit (`Activity<T>.id`) or generated by the SDK on Android. Exposed in `LiveActivityInfo.activityId` for local ActivityKit correlation. Not used in Customer.io server routes — all instance-level routes use `activityInstanceId` instead. |
| Activity Type | A reverse-DNS string identifier registered with the SDK (e.g. `io.customer.liveactivities.livescore`). |
| User ID | The identifier the host app supplied to `CustomerIO.identify()`. |
| Installation ID | A stable UUID generated by the SDK on first launch and persisted for the lifetime of the app install. Uniquely identifies a device independently of the user identity or push token. Exposed as a read-only property on the root `cio` object. Included in push notification registration payloads and all Live Activity token registration calls. Whether it appears on general track/screen/identify events is an open design question — see OPEN_QUESTIONS.md. |
| `os` | Device operating system. `"ios"` or `"android"`. Determines the push payload shape the server constructs. |
| `transport` | Push delivery infrastructure. `"apns"` for direct Apple Push Notification service delivery; `"fcm"` for delivery via Firebase Cloud Messaging relay. Determines which credentials and API the server uses to dispatch the push. |
| Token format | Hex-encoded for APNs tokens; URL-safe base64 string for FCM registration tokens. |

**Valid `os` / `transport` combinations**

| `os` | `transport` | When to use |
|---|---|---|
| `"ios"` | `"apns"` | Partner's APNs credentials are configured directly with Customer.io. |
| `"ios"` | `"fcm"` | Partner uses Firebase for iOS push; APNs credentials live with Firebase. Server relays via FCM using the partner's FCM project credentials. ActivityKit-generated Live Activity tokens are still raw APNs tokens but can be relayed through FCM using Firebase's Live Activity API. |
| `"android"` | `"fcm"` | All Android devices. FCM is the only supported transport on Android. |
| `"android"` | `"apns"` | Invalid. |

All SDK surface requests carry the following headers:

| Header | Value |
|---|---|
| `Authorization` | `Basic <base64("cdpApiKey:")>` — the `cdpApiKey` followed by a colon, Base64-encoded. |

**Response**

```json

```

### `PUT /v1/live_activities/registration/{activity_type}`

Registers this installation's capability to receive live activities or live
updates of the given activity type. Behaviour differs by platform:

**iOS:** Called whenever ActivityKit vends a new push-to-start token for a
registered type. The token is required and stored server-side so the server can
remotely launch a new activity instance. The server stores the most recent token
per `(userId, installationId, activityType)` triple; older tokens are silently
replaced.

**Android:** Called at configure time for each activity type registered in the
SDK configuration. No token is included — the FCM registration token is already
on file from push registration and the server resolves it via `installationId`.
The call records `(userId, installationId, activityType)` so the server can
route partner-initiated start requests only to installations that support the
requested template, and correctly handle template versioning across app updates.

**Path parameters**

| Parameter | Type | Description |
|---|---|---|
| `activity_type` | string | The reverse-DNS activity type identifier. |

**Request body**

```json
{
  "token": "<hex-encoded push-to-start token>",
  "os": "ios",
  "transport": "apns",
  "userId": "usr_123",
  "installationId": "<stable per-install UUID>"
}
```

| Field | Required | Description |
|---|---|---|
| `token` | iOS only | Hex-encoded APNs push-to-start token. Always an APNs token — ActivityKit generates this directly regardless of the partner's push configuration. Omitted for Android. |
| `os` | Yes | `"ios"` or `"android"`. Tells the server whether a token is expected and which delivery infrastructure to use. |
| `transport` | Yes | `"apns"` or `"fcm"`. For iOS, indicates whether to dispatch push-to-start directly via APNs or relay through Firebase. For Android, always `"fcm"`. |
| `userId` | Yes | The currently identified user. |
| `installationId` | Yes | The stable per-install UUID for this device. |

**Response — 200 OK**

```json
{ "registered": true }
```

**Response — 400 Bad Request**

```json
{ "error": "<missing field name> is required" }
```

### `PUT /v1/live_activities/{activity_instance_id}/push_token`

Registers or rotates the **instance push token** for a specific live activity.
This is the token the server uses to send content-state updates and end events
to a single running activity instance.

On iOS this is the ActivityKit instance token, distinct from both the device's
regular push token and the push-to-start token. On Android this is the
standard FCM registration token. The `os` and `transport` fields tell the
server what payload to construct and which delivery infrastructure to use.

The SDK calls this endpoint when an activity starts (initial token) and
whenever the token rotates for a running activity.

**Path parameters**

| Parameter | Type | Description |
|---|---|---|
| `activity_instance_id` | string | The partner-supplied activity identifier (`CIOActivityAttribute.activityInstanceId`), percent-encoded. |

**Request body**

```json
{
  "token": "<push token>",
  "activity_type": "io.customer.liveactivities.livescore",
  "os": "ios",
  "transport": "apns",
  "userId": "usr_123",
  "installationId": "<stable per-install UUID>"
}
```

| Field | Required | Description |
|---|---|---|
| `token` | Yes | The push token for this activity instance. Hex-encoded APNs token for iOS (always — ActivityKit generates APNs tokens regardless of the partner's push configuration); base64 FCM registration token for Android. |
| `activity_type` | Yes | Reverse-DNS identifier of the template. Used to partition tokens by template on the server. |
| `os` | Yes | `"ios"` or `"android"`. Determines the push payload shape the server constructs for subsequent updates. |
| `transport` | Yes | `"apns"` or `"fcm"`. Determines which delivery infrastructure the server uses. See valid combinations in [Common Conventions](#common-conventions). |
| `userId` | Yes | The currently identified user. |
| `installationId` | Yes | The stable per-install UUID for this device. |

**Response — 200 OK**

```json
{ "registered": true }
```

**Response — 400 Bad Request**

```json
{ "error": "<missing field name> is required" }
```

## Partner Surface

Partners interact with live activities through user identifiers. The server
maintains the mapping from user identifiers to platform push tokens
internally; partners never handle tokens directly, and platform differences
(APNs vs FCM) are transparent to the partner.

All partner endpoints require `userId` and `activityType`. An optional
`activityInstanceId` can be supplied to narrow the operation to a single running
activity when a user has multiple activities of the same type. When `activityInstanceId`
is omitted, the operation applies to all matching activities for that user and
type across all platforms.

---

### `POST /v1/users/{userId}/live_activities/{activity_type}/update`

Sends a content-state **update** to one or all running activities of the given
type for this user. The server dispatches via APNs for iOS activities and via
FCM for Android activities; the partner payload is identical in both cases.

**Path parameters**

| Parameter | Type | Description |
|---|---|---|
| `userId` | string | The user identifier supplied to `CustomerIO.identify()`. |
| `activity_type` | string | The reverse-DNS activity type identifier. |

**Request body**

```json
{
  "activityInstanceId": "optional-activity-instance-id",
  "installationId": "optional-installation-id",
  "contentState": {
    "<template-specific content-state fields>"
  }
}
```

| Field | Required | Description |
|---|---|---|
| `activityInstanceId` | No | Narrows the update to the activity instance with this `activityInstanceId`. |
| `installationId` | No | Narrows the update to a specific installation. |
| `contentState` | Yes | The new content state. Schema is template-specific (see [Content State Schemas](#content-state-schemas)). |

When neither `activityInstanceId` nor `installationId` is provided, all matching activities for this user and type receive the update. When one is provided, it is used as a filter. When both are provided, the target must match both — the update is sent only to activities that satisfy all supplied constraints.

**Response — 200 OK**

```json
{
  "sent": 2,
  "failed": 0,
  "errors": []
}
```

| Field | Type | Description |
|---|---|---|
| `sent` | int | Number of activity instances the update was delivered to (APNs or FCM returned success). |
| `failed` | int | Number of delivery failures. |
| `errors` | array | Detail for each failure. |

### `POST /v1/users/{userId}/live_activities/{activity_type}/end`

Sends an **end** event to one or all running activities of the given type for
this user, then removes their tokens from the registry.

On iOS the APNs end event causes the system to dismiss the activity from the
Lock Screen after `dismissalDate`. On Android the SDK processes the FCM data
message and cancels the ongoing notification.

**Path parameters**

| Parameter | Type | Description |
|---|---|---|
| `userId` | string | The user identifier. |
| `activity_type` | string | The reverse-DNS activity type identifier. |

**Request body**

```json
{
  "activityInstanceId": "optional-activity-instance-id",
  "installationId": "optional-installation-id",
  "contentState": {
    "<final content-state fields>"
  },
  "dismissalDate": 1713100030
}
```

| Field | Required | Description |
|---|---|---|
| `activityInstanceId` | No | Narrows the end request to the activity instance with this `activityInstanceId`. |
| `installationId` | No | Narrows the end request to a specific installation. |
| `contentState` | No | Final content state shown before the activity is dismissed. Defaults to the last-known state. |
| `dismissalDate` | No | **iOS only.** Unix timestamp after which the system removes the activity from the Lock Screen. Defaults to `now + 30 s`. |

When neither `activityInstanceId` nor `installationId` is provided, all matching activities for this user and type are ended. When one is provided, it is used as a filter. When both are provided, the target must match both — only activities satisfying all supplied constraints are ended.

**Response — 200 OK**

Same shape as the update response, with an additional `"purged"` count
reflecting how many tokens were removed from the registry.

### `POST /v1/users/{userId}/live_activities/{activity_type}/start`

Requests that a new live activity be started for this user.

#### iOS — two start paths

**Preferred path — silent push.** The server sends a silent (background) push
to the user's registered device token via the device's configured transport
(`apns` or `fcm`). Both APNs and Firebase support `content-available: 1`
background pushes; the behavior on the device is identical regardless of which
transport is used. The app wakes in the background, creates the activity locally
via ActivityKit, and the resulting instance push token flows back to the server
via the SDK surface. The partner receives the `activityInstanceId` via their own
webhook or callback mechanism.

The silent push path requires the integrated push backend to have the device's
regular push token on record alongside `installationId` and `transport`, so the
server can resolve `userId + installationId → (token, transport)` and dispatch
correctly. This join is not available in the standalone POC server; see the POC
note below.

**Fallback path — push-to-start.** When `installationId` is provided, the
server uses the APNs push-to-start token registered for that
`(userId, installationId, activityType)` triple to remotely launch the
activity without the app needing to be in the foreground. If `installationId`
is omitted, the server fans out to all push-to-start tokens registered for
this user and type. Push-to-start delivery is best-effort — iOS may not grant
background execution time, and the push-to-start token may not be registered
if the user has not opened the app recently.

Push-to-start is the appropriate fallback when the device's regular push token
is stale or absent — for example, when a user has not opened the app recently
enough for the silent push path to be viable.

#### Android — in-app start only

Android has no push-to-start equivalent. The server sends a high-priority FCM
data message signalling the SDK to create a live update. The system will wake
the app from the background to process the message, but **a force-stopped app
will not be woken**. The silent push path is the only supported start mechanism
on Android, and it carries the same dependency on the integrated push backend
having `installationId` on record.

#### POC server limitation

The standalone POC server does not have access to the regular APNs or FCM
device tokens used for the silent push start path — those are held by the main
push notification backend. As a result, the POC server exercises the
**push-to-start path only** for iOS start requests and cannot exercise the
Android start path at all. This is an accepted POC gap. The production
implementation will use the silent push path as primary once the live activity
server is integrated into the push backend and `installationId` is available
on the existing device token records.

## Push Payload Structures

The server constructs platform-specific push payloads internally. This section
is informational — partners do not send push payloads directly.

### APNs — update / end (iOS)

```json
{
  "aps": {
    "timestamp": 1713100000,
    "event": "update",
    "content-state": {
      "<template-specific content-state fields>"
    },
    "alert": {
      "title": "Optional alert title",
      "body": "Optional alert body"
    },
    "dismissal-date": 1713100030
  }
}
```

`event` is `"update"` or `"end"`. `alert` and `dismissal-date` are only
included when explicitly supplied by the partner.

### APNs — push-to-start (iOS only)

```json
{
  "aps": {
    "timestamp": 1713100000,
    "event": "start",
    "attributes-type": "io.customer.liveactivities.livescore",
    "attributes": {
      "<template-specific static attribute fields>"
    },
    "content-state": {
      "<template-specific initial content-state fields>"
    }
  }
}
```

### FCM data message — update / end / start (Android)

```json
{
  "message": {
    "token": "<fcm registration token>",
    "data": {
      "event": "update",
      "activity_instance_id": "<activity-instance-id>",
      "activity_type": "io.customer.liveactivities.livescore",
      "content_state": "<JSON-encoded content-state object>"
    }
  }
}
```

`event` is `"start"`, `"update"`, or `"end"`. For `"start"`, `data` also
includes `"attributes"` (JSON-encoded static attribute object). The SDK
decodes `content_state` (and `attributes` on start), applies the update to
the ongoing notification via `NotificationManager.notify()`, and calls the
appropriate ActivityKit equivalent to dismiss on `"end"`.
